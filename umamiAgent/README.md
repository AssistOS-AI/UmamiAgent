# Umami Agent

`umamiAgent` exposes self-hosted Umami data through controlled Ploinky MCP tools.

## Components

- Ploinky AgentServer serves `/mcp`.
- `mcp-config.json` exposes one tool per allowed Umami operation.
- `tools/umami_tool.mjs` calls the internal `MadsNyl/umami-mcp` HTTP MCP server.
- The manifest pins the custom `docker.io/assistos/umami-agent` image by immutable digest. It contains Umami, PostgreSQL, Bun, and the built `MadsNyl/umami-mcp` server.
- The manifest uses Ploinky's isolated `mode: "default"` network and declares no shared attachment or DNS alias.

## Umami Stack

`umamiAgent` is the only Ploinky agent in the Umami stack. Its startup supervisor initializes PostgreSQL, starts Umami, starts the internal MCP adapter, starts the narrow telemetry proxy, proves their readiness, and then starts Ploinky AgentServer.

The Umami dashboard/API listens on private TCP `3000` under its verified base path:

```text
http://127.0.0.1:3000/services/umami/
```

The manifest declares two separate Router targets:

| Service | Private target | Access | Surface |
| --- | ---: | --- | --- |
| `umami-dashboard` | 3000 | authenticated | Dashboard, base-path assets, navigation, and APIs under `/services/umami/` |
| `umami-telemetry` | 3001 | scoped guest | Only tracker, ingestion, and bounded preflight under `/public-services/umami-telemetry/` |

Neither target adds a physical-host publication. The dashboard retains Umami's
own authentication behind Ploinky authentication. The `3001` proxy strips
cookies, authorization, forwarding, and Router identity/invocation headers
before it calls process-local Umami `3000`; it enforces exact Origins, content
type and body limits, deadlines, and rate limits.

Every ingestion request must carry the Router-owned
`x-ploinky-rate-source`. The Router removes client-supplied `x-ploinky-*`
headers and generates one route-scoped 64-hex partition from trusted canonical
transport-source material after route and policy validation. The proxy requires
that exact shape, canonicalizes hex case, uses it for the per-source bucket, and
removes it before forwarding. It never substitutes the shared Router socket
peer, an allowed Origin, a guest session, or a user identity. Missing or
malformed source partitions fail before Umami is dialed.

| Telemetry setting | Default | Contract |
| --- | ---: | --- |
| `UMAMI_TELEMETRY_PER_SOURCE_PER_MINUTE` | 120 | Maximum admitted ingestion requests for one Router-derived source partition per fixed minute |
| `UMAMI_TELEMETRY_GLOBAL_PER_MINUTE` | 1200 | Maximum admitted ingestion requests across all source partitions per fixed minute |

The retired per-Origin setting has no compatibility alias or fallback.

PostgreSQL data persists in the agent root storage at `/root/postgres`, which Ploinky maps to the workspace `.data` area. Database and app secrets are generated by Ploinky through `UMAMI_POSTGRES_PASSWORD` and `UMAMI_APP_SECRET`.

Runtime contract v5 does not read, translate, migrate, or automatically delete
retired state. Any destructive operator prerequisite must be completed before
v5 activation, outside the runtime.

Website tracking snippets use only the active `umami-telemetry` Router locator.
They never send browser traffic directly to dashboard TCP `3000` or MCP.

## IDE Settings Plugin

`IDE-plugins/umami-settings/` contributes the `Umami Settings` entry in AchillesIDE. On each launch it resolves one current browser locator at a time from Ploinky's authenticated, `no-store` topology projection. It never fabricates `.localhost` URLs or caches a startup URL. The settings modal generates a tracking script from:

- the active scoped telemetry locator; and
- the Umami website selected from the Umami MCP website list.

The plugin calls `umami_websites_list` through `/umamiAgent/mcp` when the modal opens and shows the returned websites in a selector when Umami credentials are configured. If a new website is added in the Umami dashboard, close and reopen the settings modal to reload the list. Errors from the MCP call are displayed in the modal and logged with `console.error`. Tracking data still goes directly from the website browser to Umami's `/script.js` endpoint, not to MCP.

## MCP Backend

The custom image is built only from the source lock in
`container-image-builds/images/umami-agent/sources.lock.json`: Umami `3.2.0`
uses an immutable amd64/arm64 OCI index, Bun `1.3.14` uses architecture-specific
musl archives verified by SHA-256, and `MadsNyl/umami-mcp` is checked out and
verified at exact commit `3ab73beda2db0ebffb0b07439b218ef562107520` before a
frozen-lockfile build into `/opt/umami-mcp`. There is no mutable source override
or installer-script path. The lock is embedded read-only in the image and its
critical values are also image labels. `scripts/start-umami-agent.sh` starts
that server on `127.0.0.1:${UMAMI_MCP_PORT:-7301}` before launching Ploinky
AgentServer.

The wrapper authenticates to the explicitly private internal MCP bind with OAuth using:

- `UMAMI_BASE_URL`
- `UMAMI_USERNAME`
- required generated/operator `UMAMI_PASSWORD`
- `MCP_SECRET`
- `OAUTH_CLIENT_ID`
- `OAUTH_REDIRECT_URI`

Startup fails closed when `UMAMI_PASSWORD` or exact
`UMAMI_TELEMETRY_ALLOWED_ORIGINS` is absent. There is no demonstration-password,
loopback-origin, or inferred-host fallback.

## Exposed Tools

- `umami_websites_list`
- `umami_stats_get`
- `umami_pageviews_get`
- `umami_metrics_get`
- `umami_events_list`
- `umami_active_get`
- `umami_sessions_get`
- `umami_report_generate`

The agent intentionally exposes no Umami write, admin, user, team, website CRUD, or event ingestion operations.
